## 🎯 Contexto y Problema

Necesitamos construir un sistema **standalone, client-side only** para generar videos publicitarios 10-30s (modelo AIDA) para RRSS (Reels, TikTok, Shorts, Feed) a partir de:
- Brief dinámico universal (negocio, servicios, visión global, 4 etapas AIDA)
- Keyframes reales del negocio (logo, fotos problema/espacio/solución/cta)
- Pipeline generativo: Imagen 3 (keyframes) → Veo 3.1 I2V (transiciones keyframe→keyframe) → Gemini TTS → FFmpeg WASM (concat + burn subs + mix audio)

**Restricciones críticas:**
1. **Cero backend propio** — Sin servidor, base de datos, auth, facturación
2. **API Keys seguras** — Nunca en cliente, ni en localStorage
3. **Anti-alucinación** — Videos anclados a realidad visual del negocio
4. **Control humano total** — Prompt Gates obligatorios, edición granular
5. **Persistencia local** — Sobrevive a refresh, offline-first
6. **Extensible** — Arquitectura multi-provider preparada (aunque S1 solo Gemini)

## 🏛️ Decisiones Arquitectónicas

### 1. Stack Tecnológico Definitivo

| Capa | Tecnología | Justificación |
|------|------------|---------------|
| **Build/Dev** | Vite 5 + TypeScript 5.5 | HMR rápido, ESM nativo, typecheck estricto |
| **UI Framework** | React 18 | Componentes reutilizables, testing, ecosistema |
| **Styling** | Tailwind 3.4 + CSS Custom | Carbon dark theme, responsive, tree-shaking |
| **Estado Global** | Zustand + Middleware `persist` | Reactivo, ligero, devtools, serializable |
| **Persistencia** | IndexedDB (`idb` wrapper) | Offline-first, blobs grandes, transaccional |
| **Video Generativo** | Gemini API (Imagen 3, Veo 3.1, TTS, Vision) | Un proveedor, calidad probada, pricing conocido |
| **Video Processing** | `@ffmpeg/ffmpeg` WASM (WebWorker) | Client-side concat, burn subs, mix audio, multi-ratio |
| **Proxy API Keys** | Cloudflare Workers | Serverless, edge, secrets seguros, rate limiting, CORS |
| **Testing** | Vitest + Playwright + axe-core | Unit, E2E, Accessibility en CI |
| **Deploy** | Vercel/Netlify (frontend) + Cloudflare (worker) | Preview deploys, custom domain, headers seguros |

### 2. Patrón Keyframe Chain (Anti-Alucinación Core)

**Problema:** Veo T2V/I2V con solo prompt → alucina entornos, objetos, personas no reales.

**Solución:** **Keyframe Chain System** — Cada transición es I2V entre **dos anclas visuales reales**:

```
KF0 (Logo real) ──I2I──▶ KF1_IN (Foto problema real) ──I2V──▶ KF1_OUT (Generada Imagen 3 desde KF1_IN + intención)
     │                                                                    │
     ▼                                                                    ▼
KF2_IN (Foto taller real) ◀──I2V── KF2_OUT (Generada) ◀──I2V── KF3_IN (Foto solución real) ...
```

**Reglas inary...
```

**Garantías:**
- Frame 0 de cada clip = imagen real subida por usuario O generada por Imagen 3 desde imagen real anterior
- Frame final = imagen real siguiente O generada desde real anterior
- Veo **solo interpola** entre dos anclas visuales conocidas
- Prompt incluye **análisis visual exhaustivo** de ambas anclas (Gemini Vision) + intención humana + specs cámara

### 3. Prompt Approval Gates (Human-in-the-Loop Obligatorio)

**Ninguna llamada generativa (Veo, Imagen 3, TTS) se ejecuta sin:**

1. Sistema construye prompt completo (anclas + intención + cámara + estilo + reglas anti-alucinación)
2. UI muestra **Prompt Gate**: textarea monospace + miniaturas anclas IN/OUT + diff vs original
3. Usuario **edita directamente** el prompt si desea
4. Usuario click **"APROBAR Y GENERAR"** → **solo entonces** se dispara API call
5. Sistema guarda: prompt original, prompt aprobado, diff, timestamp, usuario → `manifest.json`

### 4. Proxy Backend para API Keys (Seguridad Obligatoria)

**Problema:** `localStorage` = XSS roba keys. Variables de build (`VITE_`) = expuestas en bundle.

**Solución:** **Cloudflare Worker Proxy**

```
Cliente (Browser)                    Cloudflare Worker                     Gemini API
     │                                      │                                  │
     ├── POST /api/gemini/generateContent ──▶│                                  │
     │                                      ├── Inyecta Authorization: Bearer $GEMINI_KEY
     │                                      ├── Rate limit 10 req/min/IP
     │                                      ├── CORS headers
     │                                      ├── Log request/response (sin keys)
     │                                      ▼                                  ▼
     │                                      ◀─── Response ───────────────────────
     ◀──────────────────────────────────────│                                  │
     │                                      │                                  │
```

**Cliente NUNCA ve la key.** Worker secret `$GEMINI_API_KEY` en Cloudflare Dashboard.

### 5. Persistencia Dual: Zustand + IndexedDB (Offline-First)

```typescript
// stores/projectStore.ts
const useProjectStore = create<ProjectState>()(
  persist(
    (set, get) => ({ /* ... actions ... */ }),
    {
      name: 'bridge-project',
      storage: createJSONStorage(() => idbStorage), // IndexedDB real
      partialize: (state) => ({
        brief: state.brief,
        keyframes: Array.from(state.keyframes.entries()),
        transitions: Array.from(state.transitions.entries()),
        clips: Array.from(state.clips.entries()), // Blobs serializados
        manifest: state.manifest,
        jobQueue: state.jobQueue,
        version: PROJECT_STORE_VERSION
      }),
      onRehydrateStorage: () => (state) => {
        // Migración de versiones, validación de integridad
        migrateIfNeeded(state);
      }
    }
  )
);
```

**Ventajas:** Recarga instantánea (hidratación < 100ms), blobs videos persisten, survive refresh/crash, sin servidor.

### 6. FFmpeg WASM en WebWorker (Non-Blocking)

```typescript
// workers/ffmpeg.worker.ts
const ffmpeg = createFFmpeg({ 
  corePath: '/ffmpeg-core/ffmpeg-core.js',
  log: true 
});

self.onmessage = async (e) => {
  const { type, payload } = e.data;
  switch (type) {
    case 'INIT': await ffmpeg.load(); break;
    case 'CONCAT': 
      await writeFiles(payload.files);
      await ffmpeg.run(...payload.args);
      const data = ffmpeg.FS('readFile', payload.output);
      self.postMessage({ type: 'RESULT', payload: new Blob([data]) });
      break;
    case 'SMART_CONCAT': /* input mapping dinámico */ break;
  }
};
```

**Por qué WebWorker:** Main thread libre para UI, encoding 30-60s no bloquea interacción, memory isolated.

### 7. Background Job Queue + Service Worker (Survive Refresh)

**Problema:** Veo tarda 2-5 min/clip. Usuario cierra pestaña → pierde todo.

**Solución:** 
- `jobQueue` en IndexedDB (persistente)
- Service Worker registra `backgroundfetch` o `periodicsync` para reanudar
- Al cargar app: `jobStore.hydrate()` → reanuda jobs pendientes
- Notificación `Notification API` al completar: "Tu video está listo"

## ❌ Qué NO Hacemos (Exclusiones Explícitas)

| Exclusión | Razón |
|-----------|-------|
| Backend propio (Node/Python/Go) | Complejidad, costo, mantenimiento, single-point-of-failure |
| Base de datos (PostgreSQL/Mongo/Supabase) | Datos son del usuario, locales, privados |
| Auth (Clerk/Auth0/Firebase Auth) | Sin cuentas, sin usuarios, sin PII |
| Facturación/Stripe | Usuario paga keys directamente a Google Cloud |
| WebSockets / Real-time server | Polling HTTP suficiente para Veo operations |
| CDN propio para assets | Blobs en IndexedDB + File System Access API |
| Multi-tenant / SaaS billing | Herramienta local, no producto multi-usuario (yet) |
| Colaboración tiempo real | Single-user, single-session |
